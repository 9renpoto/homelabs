## Official Installation Methods

### 1. One-liner (Recommended - Production)
```bash
curl -fsSL https://www.nvidia.com/nemoclaw.sh | bash
```

**What it does:**
- Auto-installs Node.js 22.16+ via nvm (user-local)
- Installs `nemoclaw` CLI via npm (user-local)
- Runs `nemoclaw onboard` (interactive setup wizard)
- Creates OpenShell gateway + sandbox
- Configures Privacy Policies & Network Policies
- Auto-detects Ollama for local inference

**User runs after install:**
```bash
nemoclaw <name> connect          # Enter sandbox
openclaw tui                      # Launch agent UI
openclaw agent --agent main --local -m "hello"  # Or send single message
```

### 2. npm Install (If Node.js already installed)
```bash
npm install -g git+https://github.com/NVIDIA/NemoClaw.git
nemoclaw onboard
```

### 3. Clone & Build (For developers)
```bash
git clone https://github.com/NVIDIA/NemoClaw.git
cd NemoClaw && sudo npm install -g .
nemoclaw onboard
```

## What `nemoclaw onboard` Does

Interactive setup wizard that:
1. Creates sandbox name
2. Selects inference model:
   - Nemotron 3 Super 120B (local, full privacy, ~48GB VRAM for single GPU)
   - Nemotron 3 Nano 4B (edge, ~8GB VRAM)
   - Cloud models (OpenAI, Anthropic, NVIDIA Endpoints)
3. Configures Privacy Policies (which data stays local vs routes to cloud)
4. Sets Network Policies (default deny-all, operator approves domains)
5. Tests sandbox with sample agent

**Output:**
- `~/.nemoclaw/config.yaml` — persisted configuration
- OpenShell gateway running as systemd user service (or equivalent)
- NemoClaw-managed sandbox created

## Key Architectural Differences: Official vs Current Implementation

| Aspect | Official | Current Ansible |
|--------|----------|-----------------|
| CLI Layer | Node.js nemoclaw CLI (user-local) | Docker container |
| OpenShell | nemoclaw CLI manages gateway | Not explicitly managed |
| Config Storage | ~/.nemoclaw/config.yaml | Environment vars in Docker |
| Service Management | systemd (user-level) | Docker daemon |
| Update Path | `npm update -g nemoclaw` | Docker image pull |
| Dependency Updates | Dependabot-friendly (package.json) | Manual image tag updates |

## Critical Observation: config.yaml

The `~/.nemoclaw/config.yaml` file is the **single source of truth** after `nemoclaw onboard`.

**What it likely contains:**
- Sandbox name & ID
- Selected model(s) (Nemotron/Nano/Cloud)
- Privacy policy rules
- Network policy rules (allow/deny domains)
- Inference routing configuration

**Implication for Ansible:**
- If config.yaml is purely static after onboard → Can template it in Ansible
- If config.yaml requires interactive choices → Must run `nemoclaw onboard` manually or automate it
- **Key question:** Can config.yaml be generated from YAML template + CLI flags? (Dependabot-friendly)

## WSL2-Specific Considerations (from docs)

**Official docs note:**
- Container runtime: Docker Desktop with WSL backend (tested with limitations)
- Requires WSL2 (not WSL1)
- Allocate sufficient memory in .wslconfig (min 8GB)
- GPU passthrough requires Windows 11 22H2+

**GPU setup:**
- NVIDIA drivers 550+ on Windows
- nvidia-container-toolkit for Docker GPU access
- `nvidia-smi` visible inside WSL2 and Docker

## Decision Points for Current Implementation

### Option A: Keep Docker-first approach (Current)
**Pros:**
- Simpler Ansible orchestration
- Self-contained (no host-level Node.js)
- Clear role separation

**Cons:**
- Diverges from official recommended path
- May miss nemoclaw CLI features/updates
- config.yaml not accessible from host

### Option B: Switch to CLI-first (Official)
**Pros:**
- Aligns with official recommended path
- config.yaml in ~/.nemoclaw (host-accessible)
- Dependabot can track nemoclaw version in package.json
- Direct CLI updates via `npm update -g`

**Cons:**
- Requires Node.js on host (WSL2)
- Ansible role becomes simpler (mostly orchestration)
- Must accommodate `nemoclaw onboard` interactivity

### Option C: Hybrid (Ansible prep for manual nemoclaw onboard)
**Pros:**
- Ansible handles pre-flight checks + Docker/GPU setup
- User runs `nemoclaw onboard` manually once
- config.yaml generated by official CLI (most accurate)
- Preserves reproducibility

**Cons:**
- Requires manual step after Ansible
- Less fully automated

## Recommendation

**Next step:** Run `nemoclaw onboard` locally (outside Ansible), capture the exact contents of `~/.nemoclaw/config.yaml`, then decide:

1. If config.yaml is **simple & static** → Option A (Docker-first, template config.yaml in Ansible)
2. If config.yaml is **complex or runtime-dependent** → Option B (CLI-first, Ansible sets up Node.js + runs nemoclaw onboard)
3. If a **balanced approach** works → Option C (Ansible + manual onboard)

**Goal:** Enable dependabot version tracking & maintain reproducibility.
